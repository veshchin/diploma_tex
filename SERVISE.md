# SERVISE.md

Этот файл описывает сервисный контур проекта, контейнеризацию,
Kubernetes-инфраструктуру и CI/CD на основе `описание диплома.md`.
Имя файла сохранено в форме `SERVISE.md`, потому что именно так он был
запрошен пользователем.

## Назначение сервисного слоя

Сервисный слой нужен для интеграции вычислительного ядра во внешнюю
инфраструктуру.
В отличие от desktop-приложений, здесь основной сценарий связан не с ручным
вводом, а с пакетной обработкой большого числа точек и включением расчётов в
другие системы.

Базовым компонентом является `if97_calculator_service`.

## gRPC API

Сервис использует gRPC и предоставляет двунаправленный потоковый метод
`If97Service/CalculatePt`.

Протокол описан в `service/proto/if97.proto`.

Формат данных построен по схеме Struct-of-Arrays:

- вход: `PtBatchRequest { batch_id, p[], t[] }`
- выход: `PtBatchResponse { batch_id, h[], status[], batch_error_* }`

Преимущества такого подхода:

- удобно передавать крупные батчи;
- уменьшаются накладные расходы на сериализацию;
- проще организовать высокопроизводительную обработку однотипных данных.

## Модель параллелизма

Сервис разделяет I/O и CPU-нагрузку.

`tokio`

- принимает сетевые запросы;
- обслуживает gRPC runtime;
- работает с потоками ввода-вывода.

`rayon`

- выполняет CPU-bound расчёты;
- обрабатывает батчи в пуле фиксированного размера.

Такое разделение важно для производительности, потому что сетевой runtime не
должен блокироваться тяжёлыми термодинамическими вычислениями.

## Backpressure и ограничения

Для устойчивости сервиса используются ограничения на количество батчей в
обработке:

- `IF97_MAX_IN_FLIGHT_PER_CONN`
- `IF97_MAX_IN_FLIGHT_GLOBAL`

Также задаётся ограничение на размер батча:

- `IF97_MAX_BATCH_LEN`

И на размер gRPC-сообщения:

- `IF97_GRPC_MAX_MSG_BYTES`

Такая схема предотвращает неконтролируемый рост очередей и помогает держать
поведение сервиса предсказуемым под нагрузкой.

## Конфигурация через переменные окружения

Основные переменные:

- `IF97_GRPC_ADDR`
- `IF97_IO_THREADS`
- `IF97_CPU_THREADS`
- `IF97_MAX_IN_FLIGHT_PER_CONN`
- `IF97_MAX_IN_FLIGHT_GLOBAL`
- `IF97_MAX_BATCH_LEN`
- `IF97_GRPC_MAX_MSG_BYTES`
- `IF97_DRAIN_SECS`
- `IF97_KERNEL`

Для диплома полезно подчеркнуть, что сервис конфигурируется без пересборки,
что удобно для контейнерного и оркестраторного развёртывания.

## Health-check и корректное завершение

Сервис поднимает `tonic-health`.
При штатной работе он сообщает статус `Serving`.
При завершении:

- переводится в `NotServing`;
- уведомляет обработчики;
- выполняет `drain`;
- завершает активные операции корректно.

Это важно для эксплуатации под нагрузкой и особенно для Kubernetes, где
приложение должно предсказуемо переживать rolling updates и остановку pod.

## Docker

Контейнеризация реализована через `service/Dockerfile`.
Используется многостадийная сборка.

Основные идеи:

- компиляция выполняется в builder-стадии;
- в build-окружение ставится `protobuf-compiler`;
- в итоговый образ попадают только необходимые артефакты.

Преимущества:

- уменьшается размер конечного образа;
- упрощается развёртывание;
- повышается воспроизводимость поставки.

## Kubernetes

В проекте подготовлены Kubernetes-манифесты `service/k8s/*`.
Среди них:

- Deployment
- Service
- HPA
- PDB
- Job для нагрузочного тестирования

Это означает, что проект ориентирован не просто на локальный запуск
бинарника, а на эксплуатацию в оркестраторе.

Для диплома здесь важно подчеркнуть:

- возможность масштабирования;
- ограничение деградации доступности;
- наличие отдельного сценария load testing;
- пригодность сервиса к эксплуатации в облачной среде.

## Локальный запуск сервиса

Пример команды:

```bash
IF97_GRPC_ADDR=0.0.0.0:50051 cargo run -p if97_calculator_service --bin if97_calculator_service
```

## Верификация и нагрузочное тестирование

Сервисный слой проверяется не только функционально, но и по нагрузочным
сценариям.
Для этого используются:

- тесты ядра, от которого зависит сервис;
- `grpc_loadtest`;
- Kubernetes Job для нагрузочного прогона.

Это важно для диплома, потому что демонстрирует не только корректность
алгоритма, но и готовность к эксплуатационной нагрузке.

## CI/CD

Автоматизация построена на GitHub Actions.

`docs.yml`

- собирает rustdoc;
- публикует документацию на GitHub Pages.

`release.yml`

- запускается по тегам формата `vMAJOR.MINOR.PATCH`;
- выполняет валидацию workspace;
- гоняет тесты ядра;
- собирает FLTK и Tauri bundles;
- собирает web-артефакты;
- публикует артефакты релиза.

Инженерные детали релизной автоматизации (из родмапы проекта):

- сборки выполняются под Windows, Linux и macOS;
- для части Windows-сценариев применялась кросс-компиляция (например, через `cargo-xwin`);
- решались практические вопросы окружения (версии Node.js для Tauri/web-части,
  кэширование и абсолютные пути сборки);
- в релизы попадают как установочные пакеты/бандлы Tauri, так и портативные артефакты
  (включая FLTK-bundled сборки, не требующие установки внешних зависимостей).

Для диплома стоит отдельно отметить:

- семантическое версионирование;
- автоматическую сборку на нескольких ОС;
- автоматическую публикацию документации;
- автоматическую публикацию релизных бинарных артефактов.

## Что важно подчеркнуть в дипломе

- сервисный слой превращает вычислительное ядро в интеграционный компонент;
- gRPC и SoA-формат ориентированы на пакетные высокопроизводительные расчёты;
- backpressure и лимиты являются частью инженерной надёжности;
- Docker и Kubernetes показывают пригодность проекта к современной
  инфраструктуре;
- CI/CD завершает контур от разработки до поставки артефактов.
