# Проектные правила (канон)

Этот файл фиксирует рабочие правила и ограничения проекта.
Полная карта “что где лежит” и привязка рисунков/таблиц к файлам — в `AGENTS/PROJECT_SUMMARY.md`.

## Структура репозитория

- `source/` — редактируемые LaTeX-исходники и `references.bib`.
- `source/main.tex` — единственная точка входа итоговой сборки.
- `source/preamble.tex` — общие настройки вёрстки.
- `source/chapters/` — главы и части глав.
- `data/theory/` — теоретические/конспективные материалы (не для сборки LaTeX).
- `results/` — только готовые PDF-артефакты.
- `.build/` — временные файлы сборки, промежуточные PDF, логи и скрипты.

## Политика разбиения `.tex` (лимит строк)

Цель: каждый редактируемый `.tex` держать в пределах ~250–300 строк.

Правило:

- большие главы режутся на смысловые части `source/chapters/*_partNN.tex`;
- файл главы становится обёрткой (глава + `\input{chapters/..._partNN.tex}`), без большого текста внутри.

Если вы добавили/переставили `*_part*.tex`, обновляйте:

- `source/chapters/<chapter>.tex` — список `\input`;
- `.build/chapters/build_chapters.sh` — список компиляции PDF-фрагментов в `results/chapters/`.

## Жёсткие ограничения

- Не менять композицию документа (какие главы есть и в каком порядке) без явного подтверждения пользователя.
- Не возвращать отдельный лист «ОСНОВНАЯ ЧАСТЬ». Если где-то остались упоминания — удалять как мусор.
- Не оставлять итоговые PDF в корне проекта.
- Собирать только `XeLaTeX` или `LuaLaTeX` (для Times New Roman через `fontspec`).
- Любые правки — минимально и по существу (без “перепридумывания” текста).

## Сборка (канонично)

- Полная сборка: `.build/build.sh` → итог `results/diploma.pdf`.
- Частичные PDF (по фрагментам): `.build/chapters/build_chapters.sh` → `results/chapters/*.pdf`.
- Техническая проверка без PDF: `xelatex -no-pdf` с выводом в временный каталог (`.build/check_*`).

## Правила оформления (вёрстка)

- Ориентир: ГОСТ 7.32-2017, ГОСТ Р 2.105-2019, ГОСТ Р 7.0.100-2018 + правила кафедры.
- Абзацный отступ: 1.25 см.
- Абзацы в `.tex` по возможности начинать с новой смысловой строки.
- В списках первого уровня не использовать локальные костыли вида `\item[] -`.
- Если список уже включён через глобальные настройки, элементы: строчная буква; `;` в промежуточных; `.` в последнем.
- После формул блоки расшифровки: `где` без двоеточия и без `\noindent`, каждая переменная с новой строки.
- В таблицах/подписях не использовать жёсткое позиционирование, если уже задано семантическое решение в преамбуле.
- Математические и физические величины оформлять последовательно в математическом режиме.

## Устойчивые формулировки

- Русские названия регионов: `Регион 1`, `Регион 2`, `Регион 2 Метастабильный Пар`.
- Английские `Region`, `Region 2 Meta` и похожие варианты в тексте не использовать.
- Технические сущности оформлять семантически: `модуль \texttt{tokio}`, `сообщение \texttt{PtBatchRequest}`, `ресурс \texttt{Deployment}`, `переменная \path{IF97_MAX_IN_FLIGHT_PER_CONN}`.
- Избегать заголовков вида “(... Tokio)”, “(... Docker и Kubernetes)”: технология должна быть частью нормальной русской фразы.

## Библиография (`source/references.bib`)

- Для авторов использовать явный `author`.
- Убирать хвосты вида `/ ...` из `title`.
- Проверять `year` у `@online`/`@report`.
- Опираться на официальный источник, а не на пересказ.

## Финальная самопроверка (после правок)

- В `source/` не появилось новых “монолитных” `.tex` >300 строк (резать на `*_partNN.tex`).
- В корне проекта не появилось `main.pdf`, `Diploma.pdf`, `Taskfile.yml` и прочих артефактов.
- Новые PDF попали только в `results/` и `results/chapters/`.
- В списках не появилось пунктов, начинающихся с `\texttt{...}` или `\path{...}`.
- Не вернулся отдельный лист «ОСНОВНАЯ ЧАСТЬ».
- Не осталось английских `Region*` в русскоязычном тексте.
