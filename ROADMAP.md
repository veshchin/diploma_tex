# Roadmap разработки проекта и материалы для задания на ВКР

Этот файл фиксирует дорожную карту разработки программного комплекса `if97_calculator`
(калькулятор теплофизических свойств воды и водяного пара по стандарту IAPWS-IF97)
и содержит черновые формулировки для заполнения бланка задания на ВКР (Приложение 15).

## Кратко о проекте

- Цель: кроссплатформенный инженерный калькулятор свойств воды/пара по IAPWS-IF97 с несколькими каналами доставки.  
- Ключевой принцип: одно вычислительное ядро `if97_core` и thin-layer адаптеры поверх него.  
- Контракты обмена: общий слой DTO `if97_app_api`.  
- Каналы доставки результата: desktop (FLTK), web/desktop (Yew/WASM + Tauri), сервисный контур (gRPC), контейнерный запуск (Docker/Kubernetes).  
- Инженерные приоритеты: stateless вычисления, воспроизводимая сборка, минимизация аллокаций в горячих участках, устойчивость на границах фаз.  

## Часть 1. Этапы реализации (хронология)

### 1. Проектирование

- Архитектура: DDD-подход с разделением на доменную модель (`domain`), вычисления (`if97_core`) и thin-layer интерфейсы.  
- Организация репозитория: Rust Cargo Workspace для согласованной сборки нескольких крейтов.  
- Данные: константы и коэффициенты IF97 хранятся в CSV и преобразуются в статический Rust-код на этапе сборки через `build.rs`.  
- Типизация: newtype-единицы измерения (например, `MegaPascal`, `Kelvin`) как защита от ошибок размерности.  

### 2. Реализация ядра (`if97_core`)

- Прямые уравнения: реализованы регионы 1--5 и линия насыщения (Region 4).  
- Потенциалы: регионы 1, 2, 5 реализованы через безразмерную энергию Гиббса $\gamma$, регион 3 через энергию Гельмгольца $\phi$.  
- Обратные уравнения: внедрены аналитические обратные формулы IF97 там, где они заданы стандартом (например, $T(p,h)$ и $T(p,s)$ для регионов 1 и 2).  
- Маршрутизация: фасад `If97` определяет регион по входным параметрам и запускает корректный план вычисления.  
- Расширение маршрутов: добавлен расчёт по паре $\rho T$ как численное расширение (подбор давления методом секущих с последующим расчётом по $pT$).  

### 3. Реализация FLTK UI

- Интерфейс: desktop-приложение на `fltk-rs` с вкладками «Одиночный расчёт», «Табличный расчёт», «Графики», «О программе».  
- Пакетная обработка: парсинг табличных данных, включая сценарии из буфера обмена.  
- Графики: визуализация через `plotters` (рендер в память/буфер и вывод без мерцания).  
- Купол насыщения: построение границ $x=0$ и $x=1$ для диаграмм.  

### 4. Реализация WASM (Yew)

- Фронтенд: реактивный UI на Yew, сборка через Trunk.  
- Визуализация: диаграммы на HTML5 Canvas через `plotters-canvas`.  

### 5. Рефакторинг и стабилизация ядра

- Унификация: переход к общим трейтам для региональных моделей (`ThermodynamicRegion`, `WaterRegionModel`, `HelmholtzRegion`).  
- Производительность: кэширование степеней $\pi$ и $\tau$, сокращение дублирования, аккуратное суммирование рядов (например, через `.fold()`).  
- Устойчивость Region 3: метод Ньютона--Рафсона заменён на 2D-решатель Левенберга--Марквардта с Grid Search и демпфированием шага.  
- Топология: реализована граница B23, добавлена защита от некорректных фазовых переходов при обратных расчётах, выделен метастабильный пар (`Region2Meta`).  

### 6. WASM + Tauri

- Desktop-оболочка: web-приложение упаковано в Tauri v2 с безопасным IPC для файловых диалогов и экспорта (CSV/PNG).  
- Состояние UI: переход к «единому источнику истины» (`AppContext`, `PersistentState`).  
- Навигация: отказ от удаления узлов из DOM в пользу управления видимостью (CSS-классы).  
- Оптимизация графиков: контроль актуальности async-ответов по `req_id`, кэш купола по ключу `(chart_type, swap_axes)`.  
- Семплинг купола: адаптивное уточнение, для оси $v$ использовалась метрика $\ln(v)$ для гладкости диаграмм типа $p$--$v$ и $T$--$v$.  
- Производительность рендера: сборка фронтенда в `release`, учёт `devicePixelRatio`, устранение лишних DOM-lookup при рендере.  

### 7. Рефактор FLTK

- Архитектура: переход от `Arc<Mutex<T>>` к MVU-паттерну и обмену сообщениями через `fltk::app::channel`.  
- Графика: стабилизация масштабирования, отсечение линий (Коэн--Сазерленд), фильтрация `NaN` до передачи в рендерер.  

### 8. Релиз и CI/CD (GitHub Actions)

- CI/CD: сборки под Windows, Linux, macOS, кросс-компиляция через `cargo-xwin`.  
- Релизные артефакты: установочные пакеты Tauri и портативные (в том числе FLTK-bundled) варианты.  
- Исправления CI: установка `protoc` для Linux (требуется `tonic_build`), добавление этапов сборки rustdoc и web-артефактов.  
- Версионирование: разделение «публичной» версии тега (например, `v1.0`) и semver-ограничений в `Cargo.toml` (формат `x.y.z`).  
- Нюансы окружения: требования GitHub Actions к Node/JS-actions и особенности кэширования путей Tauri.  

## Часть 2. Ключевые решения, проблемы и метрики (коротко)

### Инженерные решения

- Изоляция предметной математики в `if97_core` и отсутствие дублирования формул в UI/сервисе.  
- Контракты обмена (`if97_app_api`) как единый формат входов/выходов между компонентами.  
- Для графиков: отдельный API насыщения (например, построение насыщенной жидкости/пара напрямую), чтобы не перегружать универсальный маршрут `px`, когда нужна только граница.  
- Для сервиса: разделение I/O и CPU (tokio vs rayon), формат данных SoA в protobuf, bidi streaming, очередь `mpsc` и backpressure.  

### Проблемы и решения

- Tauri dev не распознавал проект: выравнивание структуры `wasm/src-tauri` и корректировка dev/build-команд Trunk/Tauri.  
- Ошибка `invalid type: unit value, expected f64` в ветке $\rho T$: усиление контроля ввода и формата чисел при сериализации/десериализации.  
- «Угловатый» купол на $p$--$v$ и $T$--$v$: адаптивный семплер и метрика $\ln(v)$.  
- «Вспышка» старой кривой: защита от устаревших async-ответов по `req_id` и кэширование купола.  
- Canvas/resize: учёт `devicePixelRatio` и корректный клиппинг.  
- Переменная окружения `NO_COLOR`: требуется значение вида `NO_COLOR=true` (а не `NO_COLOR=1`) для корректного запуска некоторых CLI-сценариев.  
- CI на Linux падал без `protoc`: явная установка `protobuf-compiler` в workflow.  

### Метрики и проверки

- Внутренний ignored-тест купола: десятки миллисекунд на диаграмму.  
- Плотность точек купола (пример): $p$--$v$ около 4249 точек, $T$--$v$ около 2396 точек.  
- Нагрузочное тестирование: throughput порядка 181k rows/s (native release) и порядка 148k rows/s (в Docker).  
- Базовые проверки: `cargo test --workspace`, `cargo test -p if97_core`, сборка UI и сервисных крейтов в release.  

## Часть 3. Приложение 15 (Задание на ВКР)

Ниже сохранены формулировки для заполнения бланка (пункты 1--6).  
Сроки и проценты могут потребовать корректировки под учебный график.

### 1. Наименование темы

Разработка кроссплатформенного инженерного программного комплекса для высокоточного расчёта
теплофизических свойств воды и водяного пара по стандарту IAPWS-IF97.

### 2. Срок сдачи студентом законченной работы

(Заполняется по учебному графику.)

### 3. Техническое задание и исходные данные к работе

Исходные данные:

1. Международный стандарт IAPWS-IF97 (Industrial Formulation for the Thermodynamic Properties of Water and Steam).  
2. Наборы эмпирических коэффициентов IAPWS в формате CSV-таблиц для регионов 1--5 и метастабильного состояния.  

Техническое задание:

1. Спроектировать масштабируемую архитектуру вычислительного ядра на языке Rust.  
2. Реализовать прямые (через безразмерные энергии Гиббса и Гельмгольца) и обратные уравнения для вычисления
   удельного объёма, энтальпии, энтропии, теплоёмкости и скорости звука.  
3. Разработать алгоритмы маршрутизации, автоматического определения границ регионов и численные методы
   (Левенберга--Марквардта) для околокритической зоны (Region 3).  
4. Разработать два варианта графического интерфейса:
   - портативную нативную версию (FLTK);  
   - гибридную кроссплатформенную версию (WASM + Yew + Tauri).  
5. Обеспечить пакетную обработку данных и визуализацию термодинамических процессов на диаграммах
   ($p$--$T$, $\rho$--$T$, $T$--$s$, $h$--$s$, $p$--$h$).  
6. Настроить CI/CD процесс автоматической сборки релизов для Windows, Linux и macOS с использованием GitHub Actions.

### 5. Перечень иллюстративно-графических материалов

1. Схема архитектуры программного комплекса (Core, UI, сервис).  
2. Блок-схема алгоритма маршрутизатора (Router) для определения региона.  
3. Схема итерационного 2D-решателя для околокритической зоны (Region 3).  
4. Термодинамические диаграммы (например, $T$--$s$ и $p$--$h$) с линией насыщения и изолиниями.  
5. Скриншоты интерфейса (одиночный расчёт, табличный режим, диаграммы).
