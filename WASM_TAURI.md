# WASM_TAURI.md

Этот файл описывает связку Yew/WASM и Tauri на основе
`описание диплома.md`.
Документ нужен как опорная запись для раздела диплома о современной
desktop-оболочке с web-технологиями.

## Общая идея

В проекте реализован второй пользовательский интерфейс, основанный на
разделении frontend и backend:

- frontend строится на Yew и WebAssembly;
- desktop-упаковка и системные функции обеспечиваются Tauri;
- вычисления выполняются через то же ядро `if97_core`.

Такой подход позволяет сочетать удобство разработки UI на web-технологиях с
возможностью поставки настольного приложения.

## Web-frontend на Yew

Крейт `wasm/` содержит frontend-компоненты.
Он отвечает за:

- экраны одиночного расчёта;
- экраны табличного режима;
- экраны диаграмм;
- служебные представления, включая about и logs;
- клиентские типы и адаптеры вызовов.

Визуализация термодинамических диаграмм в web-части может выполняться напрямую
в браузере на HTML5 Canvas, например через `plotters-canvas`.
Такой подход даёт высокую производительность при интерактивных сценариях
(масштабирование, перерисовка циклов и т.п.).

Ключевая особенность текущей реализации состоит в том, что frontend не
вычисляет свойства воды самостоятельно.
Он формирует DTO и вызывает команды backend-слоя через `invoke`.

Это означает, что полноценный расчётный режим ориентирован именно на запуск
в составе Tauri-приложения.

## Роль слоя if97_app_api

Между frontend и backend используется общий набор DTO из `if97_app_api`.
Этот слой важен по нескольким причинам:

- формат обмена не размазывается по UI-коду;
- backend и frontend используют одинаковые контракты;
- упрощается сериализация и тестирование.

Примеры типов из общего API:

- `InputMode`
- `DiagramKind`
- `SingleCalcRequest`
- `TableCalcRequest`
- `TableRowResult`
- `StateDto`
- `LogEntryDto`

Для диплома стоит отдельно отметить, что `StateDto` учитывает ограничения
JSON и умеет кодировать специальные значения вроде `NaN` и `Infinity`.

## Tauri backend

Desktop-часть реализована в `wasm/src-tauri/`.
Она предоставляет набор команд `tauri::command`, вызываемых из frontend.

Основные команды:

- `calculate_single`
- `calculate_table`
- `calculate_dome`
- `load_file_dialog`
- `save_file_dialog`
- `save_plot_dialog`
- `read_logs`
- `clear_logs`

Через эти команды frontend получает доступ не только к вычислениям, но и к
системным функциям настольной ОС.

В рамках desktop-упаковки (Tauri v2) реализован безопасный IPC-мост между
frontend и ОС: диалоги открытия/сохранения CSV-файлов, экспорт графиков (PNG),
доступ к журналам и другие «нативные» функции.

## Причина использования spawn_blocking

Термодинамические расчёты являются CPU-bound задачами.
Если выполнять их непосредственно в асинхронном runtime, интерфейс может
терять отзывчивость.

Поэтому в Tauri backend тяжёлые вычисления запускаются через
`spawn_blocking`.
Это позволяет:

- не блокировать async runtime;
- отделить вычислительную нагрузку от UI и I/O;
- сохранить предсказуемую реакцию приложения на действия пользователя.

## Управление состоянием в UI

При развитии интерфейса возникала типовая проблема web-приложений: потеря
введённых данных при переключении вкладок.
Для её устранения использован принцип «единого источника истины»:
состояние приложения вынесено в общий контекст (например, `AppContext` и
`PersistentState`), а навигация переведена с удаления узлов из DOM на управление
видимостью через CSS-классы (например, `.tab-hidden`, `.tab-active`).

## Поддерживаемые пользовательские сценарии

Связка Yew/Tauri должна обеспечивать:

- одиночный расчёт;
- табличный расчёт;
- построение купола насыщения;
- визуализацию диаграмм;
- файловые операции и экспорт;
- отображение и очистку логов backend-части.

По смыслу это близко к FLTK-версии, но реализовано через другой
технологический стек и другую архитектуру взаимодействия.

## Сборка и запуск

Сборка frontend:

```bash
cd wasm
trunk build --release
```

Запуск desktop-версии в режиме разработки:

```bash
cd wasm
cargo tauri dev
```

## Почему эта часть важна для диплома

Yew/Tauri-реализация показывает, что вычислительное ядро можно встроить в
современный desktop-интерфейс, построенный на web-подходах.
Это расширяет область применения проекта и демонстрирует переносимость
архитектуры.

Для диплома полезно подчеркнуть:

- разделение UI и backend;
- использование общего DTO-слоя;
- перенос вычислений в блокирующий пул;
- сочетание кроссплатформенного интерфейса и нативных возможностей desktop.
