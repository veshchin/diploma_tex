# CORE.md

Этот файл описывает вычислительное ядро проекта на основе
`описание диплома.md`.
Документ нужен как краткая техническая карта для разделов диплома,
связанных с реализацией на Rust.

## Назначение ядра

`if97_core` является центральным компонентом всего программного комплекса.
Именно он реализует термодинамические расчёты по стандарту IAPWS-IF97.
Остальные части системы не дублируют формулы, а только адаптируют ядро под
конкретный способ использования.

Ключевая архитектурная идея проекта:

- одно вычислительное ядро;
- несколько каналов доставки результата;
- единое состояние воды на выходе;
- единая логика валидации и обработки ошибок.

При проектировании ядра использовались практики DDD: выделены доменные типы
и ошибки, а вычисления оформлены как набор детерминированных (stateless)
операций без хранения состояния между вызовами.
Инженерная цель реализации: предсказуемое поведение под нагрузкой и минимизация
аллокаций в горячих участках кода.

## Публичный API

Публичный фасад ядра представлен типом `If97`.
Основные маршруты:

- `If97::pt(p, t)`
- `If97::ph(p, h)`
- `If97::ps(p, s)`
- `If97::px(p, x)`
- `If97::rhot(rho, t)`
- `If97::metastable_pt(p, t)`

Результатом вычисления выступает агрегированное состояние `WaterState`.
В него входят:

- давление;
- температура;
- удельный объём;
- плотность;
- энтальпия;
- энтропия;
- внутренняя энергия;
- изобарная теплоёмкость;
- скорость звука;
- номер региона IF97.

## Внутренняя архитектура ядра

Ядро разбито на несколько слоёв.

`domain`

- единицы измерения;
- типы ошибок;
- итоговое состояние воды;
- типы предметной области.

`engine`

- валидация входа;
- определение региона;
- выбор плана вычисления;
- запуск нужного workflow.

`regions`

- реализация региональных уравнений IF97;
- обратные зависимости;
- численные решатели.

`topology`

- фазовые границы;
- линия насыщения;
- граница `B23`;
- классификация состояния.

`calculator`

- фасад `If97`;
- преобразование входов в `CalculationRequest`;
- унификация публичного интерфейса.

## Типобезопасность единиц

Важная особенность ядра заключается в использовании типобезопасных единиц
измерения.
Для этого применяются newtype-обёртки, например:

- `MegaPascal`
- `Kelvin`

Преимущества такого подхода:

- снижается риск перепутать размерности;
- интерфейс API становится читаемее;
- проверка корректности начинается уже на уровне типов.

## Реализация маршрутов расчёта

Маршруты `pt`, `ph`, `ps`, `px` и `metastable_pt` опираются на стандартные
региональные зависимости и обратные формулы IF97.
Маршрут `rhot` реализован как расширение с численным поиском давления,
после чего расчёт возвращается к стандартному сценарию `pt`.

Это позволяет не разносить альтернативную физическую модель по коду и
сохранять единый формат результата для всех режимов.

## Кодогенерация коэффициентов

Коэффициенты IF97 хранятся в CSV-таблицах.
На этапе сборки `core/build.rs` читает эти данные и генерирует статический
Rust-код.

Зачем это сделано:

- таблицы становятся частью бинарника;
- отсутствует runtime I/O;
- доступ к коэффициентам превращается в обращение к константам;
- сборка остаётся детерминированной.

Сгенерированные таблицы подключаются через `include!` в `tables.rs`.

## Обратные уравнения и численные решатели

Для обратных маршрутов (например, по парам $p$--$h$ и $p$--$s$) используются
аналитические обратные зависимости IAPWS там, где они определены (быстрые
формулы вида $T(p,h)$ и $T(p,s)$ для соответствующих регионов), что позволяет
избегать итераций.

Для задач, где аналитические зависимости отсутствуют или плохо устойчивы,
применяются численные методы.
При стабилизации расчётов в околокритической области (Region 3) исходный подход
на основе метода Ньютона--Рафсона оказался чувствителен к сингулярности Якобиана,
поэтому был заменён на 2D-решатель Левенберга--Марквардта с предварительным
сеточным поиском и демпфированием шага.

## Ошибки и валидация

На уровне `engine/validator.rs` проверяются:

- допустимость давления;
- допустимость температуры;
- положительность плотности;
- диапазон `x` от `0` до `1`;
- ограничение метастабильного пара.

Типичные ошибки:

- `OutOfBounds`
- `PhaseBoundaryError`
- `InvalidInput`

Для диплома важно подчеркнуть, что ошибки рассматриваются как часть
предметной логики, а не как побочный эффект вычислений.

## Роль ядра в общей архитектуре

Ядро используется:

- в FLTK desktop-приложении;
- в Yew/Tauri desktop-оболочке;
- в gRPC-сервисе;
- в тестовых и верификационных сценариях.

Таким образом, именно `if97_core` обеспечивает консистентность результатов во
всех внешних интерфейсах.

## Тестирование ядра

Проверка ядра включает:

- unit-тесты по отдельным областям и алгоритмам;
- интеграционные тесты;
- сверку с контрольными точками из CSV.

Особенно важен интеграционный сценарий `integration_csv_test.rs`, который
сравнивает расчёты с эталонными значениями.

Типовая команда проверки:

```bash
cargo test -p if97_core
```

## Рефакторинг и унификация моделей

По мере роста кодовой базы вычислительные модели были унифицированы через общие
абстракции (трейты), что позволило сократить дублирование формул и
централизовать вычисление производных.
Для производительности применяются:

- кэширование повторяющихся степеней безразмерных параметров $\pi$ и $\tau$;
- функциональные итераторы (например, `.fold()`), уменьшающие «шум» вокруг суммирования рядов;
- вынесение общих частей моделей в единые компоненты с чёткими контрактами.

Дополнительно уточнялась топология области применимости: реализовано точное
вычисление границы `B23`, добавлена защита от некорректного перехода фазы при
обратных расчётах и выделена отдельная модель метастабильного пара
(`Region2Meta`).

## Что важно подчеркнуть в дипломе

- ядро является единственным источником термодинамических вычислений;
- архитектура разделена на слои, каждый из которых отвечает за свой тип
  задач;
- типобезопасные единицы являются инженерной защитой от ошибок;
- кодогенерация коэффициентов повышает производительность и воспроизводимость;
- расширение `rhot` интегрировано в ядро без разрушения общей модели IF97.
- обратные формулы применяются там, где это возможно, а численные решатели
  используются только для действительно необходимых случаев (например, Region 3).
